# E2E 两处过程异常说明（2026-07-22）

对照全链路验证日志与代码路径，回答「ATR 偏差暂停」与「开仓瞬间核武补挂」两处疑点。

## 一、ATR 偏差暂停：触发原因与数值

### 结论（先答）

**是前者：算仓阶段独立判定的 ATR 偏差降级，不是 `-4164` 拒单「连带误触发」暂停。**

时间线证明判定发生在下单**之前**：

| UTC | 事件 | 含义 |
|-----|------|------|
| 10:57:39 | `呼吸止损: atr=14.0511 → stop=1910.52` | VPS ATR 已算出 |
| 10:57:39 | `ATR核对偏差过大: vps=14.0511 tv_implied=21.6200 Δ=53.9%` | 偏差告警 |
| 10:57:43 | `[ATR_FALLBACK] … reason=atr_mismatch_streak_3 … 本笔已用降级ATR开仓；随后暂停` | **已决定降级并挂起「成交后暂停」标志** |
| 10:57:43 | `market order failed: -4164 notional < 20` | 交易所拒单（在此之后） |
| 10:57:44 | `TRADING_PAUSED: ATR应急降级后开仓未成交·已暂停待人工确认` | 走「降级已触发但未成交仍暂停」分支 |

代码路径（`_resolve_entry_qty` → `evaluate_emergency_atr_fallback`）在下单前设置 `_atr_fallback_pending_pause=True`。  
`-4164` 只决定走了哪句暂停文案（`开仓未成交` vs 成交后的 `待人工确认`），**不参与** ATR 是否偏差的判定。

### 具体数值（第一次）

| 项 | 值 |
|----|-----|
| VPS ATR | **14.0511** |
| TV 隐含 ATR | **21.6200**（`|price−stop_loss| / TV_STOP_ATR_MULT`，mult 默认 **1.0**） |
| 偏差 | **53.9%**（相对 VPS） |
| 告警/降级阈值 | **20%**（`ATR_FALLBACK_MISMATCH_PCT` / `ATR_COMPARE_WARN_PCT`） |
| 连续次数 | **streak_3**（需连续 ≥3 次超阈值才降级） |
| 是否达阈值 | **是**（53.9% ≫ 20%，且 streak 满） |

第一次 payload 的 `stop_loss ≈ price − 1.5×ATR`。在 `TV_STOP_ATR_MULT=1.0` 下，隐含 ATR = 1.5×VPS_ATR，**必然**相对 VPS 偏大约 50%。这是测试 stop 构造与配置约定不一致导致的**真实偏差判定**，不是交易所拒单伪造的。

### 与 `-4164` 的关系（有没有不当耦合？）

| 判断 | 说明 |
|------|------|
| ATR 是否降级 | **仅看** VPS ATR vs TV 隐含 ATR；与名义不足无关 |
| 是否暂停 | 降级一旦成立，`pause_after_open=True`；**无论成交与否**都会暂停（设计如此：避免带着应急 ATR 连打） |
| `-4164` 角色 | 仅使仓位仍为空 → 命中「降级后开仓未成交仍暂停」分支 |

因此：**不是「拒单错误触发了 ATR 暂停」**；是「ATR 已决定要暂停，拒单只是成交失败的表现」。  
两类失败语义在代码里是分开的；暂停标志来自 ATR 模块，拒单来自 Binance。

**是否需要修耦合？**  
当前「降级后未成交也暂停」是有意防护，**不算误判路径**。若要优化体验，可改为：仅当降级**且成功开仓**后暂停；未成交时只告警、清 `_atr_fallback_pending_pause`、允许改 qty 重试——属产品增强，非本次 bug。

### 第二次为何同一量级 ATR 却不再暂停？

| | 第一次 | 第二次 |
|--|--------|--------|
| VPS ATR | ≈14.05 | ≈14.05（同一量级） |
| stop_loss 构造 | `price − 1.5×ATR` | `price − 1.0×ATR` |
| TV 隐含 ATR | ≈21.62 | ≈14.05 |
| 偏差 | 53.9% → 降级+暂停 | **0.0% → 不降级、不暂停** |

**不是「同一组对比数值第二次却放过」**，而是第二次改了 `stop_loss`，对比参考值变了。VPS ATR 本身没有从「坏」变「好」。

生产注意：若 Pine 的 `stop_loss` 按 1.5×ATR 画，而 VPS `TV_STOP_ATR_MULT=1.0`，会系统性抬高隐含 ATR、易累积 streak 触发降级。应对：TV 与 `TV_STOP_ATR_MULT` 约定一致，或把 mult 改为 1.5 与呼吸 `INITIAL_SL_ATR` 对齐。

## 二、开仓瞬间「核武补挂 TP」：性质

### 结论（先答）

**主要是前者：开仓后防线对齐的设计行为**（空盘口从 0/2 补到 2/2）。  
用的是与历史 TP 重复 bug **同一套** `_smart_realign_defenses` / `_nuclear_realign_tp` 子程序，但**触发条件不同**，不是超时撤单→consumed 误清那一类。

### 证据

开仓成功路径显式调用：

```text
_protect_and_monitor
  → _smart_realign_defenses(..., reason="开仓后智能防线对齐·仅TP123")
  → 审计 0/2 missing → _audit_requires_nuclear → 清场（当时无可撤）→ 挂 TP1/TP2
  → 日志「核武重挂成功: ✅TP1 … ✅TP2 …」
  → 再挂呼吸 algo stop
```

| 对比项 | 本次开仓瞬间 | 历史 TP 重复 bug |
|--------|--------------|------------------|
| 触发 | 开仓后首次审计：盘口 **0 张** TP | 超时误撤健康 TP + consumed 清空 + 周期扫描误判缺失 |
| 盘口状态 | 确实没有匹配限价单 | 往往已有/刚有匹配单却仍重挂 |
| 「重建前看盘口」修复 | 空盘 → 允许挂单；**正确生效** | 本应跳过却未跳过（已修） |
| 观察期 | ~14min **无再核武** | 连续撤挂叠单 |

因此：同一套「核武」代码，本次是 **empty-book 初始化补挂**；不是旧 bug 未覆盖边界的复发。侥幸叠单不成立——观察期稳定、平仓时仅 2 张 TP + 1 张 algo stop。

### 后续动作（按「前者」）

1. **日志措辞**：开仓 reason 含「开仓」时，不再用「核武/严重错位」恐吓用语，改为「开仓初始化补挂」等（代码已改）。  
2. **不必**为本次再开一条「重复挂单」修复单；旧修复对「盘口已有则跳过」仍然有效。  
3. （可选增强）开仓空盘时先 `_ensure_defenses_on_recover` / 短暂等待再审计，减少空转 cancel；非阻塞。
